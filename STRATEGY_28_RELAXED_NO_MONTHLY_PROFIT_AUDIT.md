# 28号不要求月月盈利审计

本文件用于定位 28号审计。它不是策略，不能交易，也不是固化版。

它只回答一个问题：

如果不要求每个月都盈利，原来的 `ret_state 64/100` 家族能不能重新优化出可用结果？

## 身份

- 审计编号：`strategy_28_relaxed_no_monthly_profit_audit_20260628`
- 来源脚本：`scripts/audit_strategy_28_relaxed_no_monthly_profit_20260628.py`
- 结果目录：`artifacts/strategy_28_relaxed_no_monthly_profit_audit_20260628/`
- 主要结果：`artifacts/strategy_28_relaxed_no_monthly_profit_audit_20260628/summary.json`
- 报告：`artifacts/strategy_28_relaxed_no_monthly_profit_audit_20260628/report.md`

## 审计口径

- 复用 14号的统一期货K线和 `ret_state 64/100` 策略家族。
- 不新增交易规则。
- 不新增市场数据。
- 不要求每个月都盈利。
- 手续费仍按开平合计 `0.2%`，代码里单边 `0.001`。
- 严格选择器每个月只能用过去月份选参数，不看未来。

## 测试方法

28号测试了 4 种严格选择方式：

- 过去亏损月份最少优先。
- 过去总收益最高优先。
- 过去总收益最高，但优先满足每月10单。
- 过去最差完整年份最好优先。

同时保留两个“看答案”的 oracle 诊断，只用来判断上限，不能交易。

## 结果

最好的严格不看未来结果仍然是：

| 项目 | 数值 |
|---|---:|
| 最好严格选择器 | `loss_control_no_positive_gate` |
| 2023 | `-21.96%` |
| 2024 | `+140.23%` |
| 2025 | `-5.28%` |
| 2026 YTD | `-0.35%` |
| 亏损月份 | `6` |
| 最差月份 | `-39.60%` |
| 最大回撤 | `-58.39%` |
| 最低月交易次数 | `10` |

看答案的月度 oracle 很强：

- 2025：`+1692.64%`
- 2026 YTD：`+171.90%`
- 但它每个月都事后挑当月最好候选，属于未来函数，不能交易。

## 判断

28号结论是：`NO_STRICT_RELAXED_UPGRADE`。

通俗说：

把“每个月都必须盈利”这个要求拿掉，也没有救回来。

看答案很好看，说明历史里确实有一些月份能靠不同参数赚到钱；但真正到当月之前，我们还是不知道该选哪一个。

## 下一步

不要继续在 `ret_state 64/100` 家族和免费K线小规则里硬挤。

如果继续研究，更现实的方向只有两个：

1. 把目标降到影子跟踪或低年化验证。
2. 等拿到真正不同的新数据源，再重新做数据质量审计和上限测试。
